# 自然语言测试用例信号匹配工具开发计划 v2

## 1. 项目目标

开发一个带前端界面的工具，支持用户上传：

1. `DBC` 文件或车企 `Excel` 信号矩阵文件，用于解析整车信号信息。
2. 自然语言测试用例文件，用于提取测试步骤与用例标识。

系统基于解析得到的信号信息和测试用例内容，调用 `DeepSeek` 大模型进行语义理解与信号匹配，最终输出结构化结果，并回填到测试用例表格中导出。

### 核心目标

1. 模型只能从提供的信号信息中查找匹配项，不能虚构信号。
2. 模型输出必须为结构化 JSON，并通过 `Pydantic` 进行标准化校验。
3. 系统需要覆盖测试用例中常见自然语言变体，包括：
   - 顺序颠倒
   - 同义词描述
   - 位置别名
   - 范围描述
   - 全量概括描述
   - 多信号拆解
   - 否定式 / 反义表达
4. 匹配结果最终可回填到原始测试用例 Excel。

---

## 2. 业务闭环

完整工作流如下：

1. 用户上传 `DBC` 或 `Excel` 信号矩阵。
2. 后端解析信号文件，生成统一信号库，返回 `signal_session_id`。
3. 用户上传自然语言测试用例 Excel。
4. 后端提取测试用例信息，返回 `case_session_id`。
5. 后端对用例描述进行规则预处理与候选信号召回。
6. 后端构建系统提示词与用户提示词。
7. 调用 `DeepSeek` API。
8. 使用 `Pydantic` 对模型输出进行强校验和标准化。
9. 将结构化结果回填测试用例表格。
10. 前端展示结果并支持导出。

## 3. 第一版范围

### 3.1 第一版必须支持

1. 信号源文件：
   - `.dbc`
   - `.xls`
   - `.xlsx`
   - `.xlsm`
2. 测试用例文件：
   - Excel 文件（`.xls` / `.xlsx` / `.xlsm`）
3. 后端技术栈：
   - `Python`
   - `FastAPI`
   - `Pydantic`
   - `SQLAlchemy 2.x`（ORM）
   - `Alembic`（数据库迁移）
   - `pydantic-settings`（环境变量管理）
4. 前端技术栈：
   - `React`
   - `Vite`
5. 模型：
   - `DeepSeek`，按 OpenAI 兼容接口封装
6. 输出：
   - 页面预览结构化匹配结果
   - 回填后的 Excel 下载
7. 数据持久化：
   - 将关键过程数据落库，便于排查问题、调整代码与优化提示词

### 3.2 第一版暂不包含

1. 用户登录
2. 历史任务管理的复杂检索与权限体系
3. 多租户
4. PDF / Word 用例导入
5. 任务队列与异步分布式执行
6. 向量检索（候选召回第一版仅用规则+关键词）

### 3.3 第一版数据库持久化要求

第一版需要加入数据库持久化，用于保留整个处理链路中的关键数据，方便后续：

1. 定位解析问题
2. 定位候选召回问题
3. 定位 Prompt 构建问题
4. 定位模型输出问题
5. 对比修改代码或提示词前后的效果变化

第一版建议至少持久化以下数据：

1. 任务基础信息
   - 任务 ID
   - 创建时间
   - 执行状态
   - 处理耗时
2. 信号源文件信息
   - 原始文件名
   - 文件类型
   - 解析结果摘要
   - 标准化信号库
3. 测试用例文件信息
   - 原始文件名
   - 用例总数
   - 解析后的用例数据
4. 语义归一化结果
   - 原始用例文本
   - 归一化文本
   - 位置展开结果
   - 否定式/反义表达转换结果
5. 候选信号召回结果
   - 每条用例对应的候选信号列表
   - 召回打分或排序信息
6. Prompt 数据
   - 系统提示词
   - 用户提示词
   - 调用参数
7. 模型响应数据
   - 原始响应文本
   - 标准化后的结构化结果
   - 校验失败信息
8. 导出结果信息
   - 回填文件路径或文件标识
   - 导出时间

第一版数据库目标不是做复杂业务系统，而是做"全过程留痕"，保证每一步都有记录可追溯。

第一版建议优先使用关系型数据库，推荐：

1. 开发阶段：`SQLite`
2. 后续可切换：`PostgreSQL`

数据库设计原则：

1. 先保证可追踪、可排查
2. 再考虑性能优化
3. 原始请求和响应尽量保留，避免问题复盘时信息缺失

### 4.1 目录说明

   放 Excel 处理、文本处理、文件处理等通用工具。

   放页面级组件。

8. `frontend/src/components/`

   暂存当前已有的 `signal_extractor.py` 与 `dbc.py`，用于迁移参考。

## 7. 后端模块详细设计

### 7.1 信号解析模块

#### 目标

将 `DBC/Excel` 信号源解析为统一信号库。

#### 输入

1. `DBC`
2. `Excel` 信号矩阵

#### 输出

统一的信号数据结构，包含：

1. 报文 ID
2. 报文名称
3. 报文长度
4. 发送节点
5. 信号名
6. 信号描述
7. 枚举值描述
8. 默认值
9. 单位
10. 周期

#### 输出示例

```json
{
  "source_file": "demo.dbc",
  "source_type": "dbc",
  "message_count": 10,
  "signal_count": 80,
  "messages": {
    "554": {
      "message_id": "554",
      "message_id_hex": "0x22a",
      "message_name": "SeatMsg",
      "signals": {
        "DrHeatSts": {
          "signal_name": "DrHeatSts",
          "comment": "主驾加热状态",
          "values": {
            "0": "OFF",
            "1": "ON"
          }
        }
      }
    }
  }
}
```

#### 后续增强

1. 输出额外的扁平化信号列表，便于候选召回。
2. 标准化报文 ID 为十进制和十六进制双格式。
3. 尽量补齐信号描述与枚举值信息。

### 7.3 文本归一化与语义增强模块

这是整个项目的关键模块之一。

#### 目标

在调用模型前，对测试用例文本做规则增强，降低模型纯字面理解带来的漏匹配和误匹配。

#### 需要覆盖的语义类型

1. 顺序颠倒
2. 同义词替换
3. 位置别名
4. 范围展开
5. 全量描述展开
6. 多对象拆解
7. 否定式 / 反义表达

#### 7.3.1 顺序颠倒

示例：

1. `打开空调`
2. `空调打开`

系统需要理解两者语义一致。

#### 7.3.2 动作同义词

归一化词典初稿：

1. 打开类：
   - 打开
   - 开启
   - 启动
   - 接通
   - 使能
   - 激活
2. 关闭类：
   - 关闭
   - 关掉
   - 关断
   - 断开
   - 停止
   - 禁用
   - 去激活

#### 7.3.3 位置同义词

第一版需覆盖：

1. `主驾 = 左前`
2. `副驾 = 右前`
3. `左后 = 后排左`
4. `右后 = 后排右`
5. `左侧 = 左前 + 左后`
6. `右侧 = 右前 + 右后`
7. `前排 = 左前 + 右前`
8. `后排 = 左后 + 右后`

#### 7.3.4 范围展开

示例：

1. `关闭全部车窗`
   展开为：
   - 主驾车窗关闭
   - 副驾车窗关闭
   - 左后车窗关闭
   - 右后车窗关闭

2. `左侧车窗打开`
   展开为：
   - 左前车窗打开
   - 左后车窗打开

第一版这里需要做得更细，不能只覆盖 `左侧/右侧/全部` 这几类表达。实际测试用例中，经常会出现各种方位、区域、相对方向、口语化位置描述，这些都需要展开成多个具体位置目标。

##### 7.3.4.1 范围展开的目标

当测试用例中的描述不是单一明确位置，而是一个区域、方向或集合时，系统需要将其展开为多个更具体的子目标，供：

1. 候选信号召回使用
2. Prompt 中的语义说明使用
3. 最终模型输出多个匹配信号使用

##### 7.3.4.2 需要覆盖的范围词类型

第一版至少覆盖以下几类范围表达：

1. 左右方向类
   - `左侧`
   - `右侧`
   - `左边`
   - `右边`
2. 前后区域类
   - `前方`
   - `前侧`
   - `前排`
   - `后方`
   - `后侧`
   - `后排`
3. 全量集合类
   - `全部`
   - `所有`
   - `整车`
   - `全车`
4. 组合区域类
   - `前部`
   - `尾部`
   - `两侧`
   - `四门`
5. 口语化位置类
   - `驾驶位这一侧`
   - `副驾这一侧`
   - `车头方向`
   - `车尾方向`

##### 7.3.4.3 范围词与位置展开规则初稿

第一版建议建立显式映射表，至少覆盖以下规则：

1. `左侧 = 左前 + 左后`
2. `左边 = 左前 + 左后`
3. `右侧 = 右前 + 右后`
4. `右边 = 右前 + 右后`
5. `前方 = 左前 + 右前`
6. `前侧 = 左前 + 右前`
7. `前排 = 左前 + 右前`
8. `后方 = 左后 + 右后`
9. `后侧 = 左后 + 右后`
10. `后排 = 左后 + 右后`
11. `两侧 = 左侧 + 右侧`
12. `四门 = 左前 + 右前 + 左后 + 右后`
13. `全部车窗 = 左前车窗 + 右前车窗 + 左后车窗 + 右后车窗`
14. `全车车窗 = 左前车窗 + 右前车窗 + 左后车窗 + 右后车窗`
15. `全部门锁 = 左前门锁 + 右前门锁 + 左后门锁 + 右后门锁`
16. `前部阅读灯 = 左前阅读灯 + 右前阅读灯`（如果信号库存在左右拆分）
17. `后部阅读灯 = 左后阅读灯 + 右后阅读灯`（如果信号库存在左右拆分）

##### 7.3.4.4 需要结合对象类型理解范围

范围展开不能脱离对象类型单独做，必须结合控制对象理解。例如：

1. `关闭全部车窗` → 展开为四个车窗相关信号
2. `关闭全部门锁` → 展开为四个门锁相关信号
3. `打开前方阅读灯` → 展开为前排或前部区域对应的阅读灯信号
4. `右边座椅加热打开` → 不能扩成车窗，必须扩成右前、右后或右侧座椅相关信号

也就是说，范围词只是位置集合，还必须和对象词结合：

1. 范围词：`右边`
2. 对象词：`车窗`
3. 动作词：`打开`

组合后才得到：

1. `右前车窗打开`
2. `右后车窗打开`

##### 7.3.4.5 典型展开示例

**示例 1：左右方向**

输入：`左侧车窗打开`

展开结果：
1. `左前车窗打开`
2. `左后车窗打开`

**示例 2：前后区域**

输入：`前方车窗关闭`

展开结果：
1. `左前车窗关闭`
2. `右前车窗关闭`

**示例 3：口语化左右**

输入：`右边座椅加热打开`

展开结果：
1. `右前座椅加热打开`
2. `右后座椅加热打开`

如果当前车型或信号库只有前排座椅加热，则后续匹配阶段应只保留存在于信号库中的合理候选。

**示例 4：全量对象**

输入：`关闭全部车窗`

展开结果：
1. `左前车窗关闭`
2. `右前车窗关闭`
3. `左后车窗关闭`
4. `右后车窗关闭`

**示例 5：全量门锁**

输入：`锁止全车车门`

展开结果：
1. `左前门锁锁止`
2. `右前门锁锁止`
3. `左后门锁锁止`
4. `右后门锁锁止`

**示例 6：否定式与范围词叠加**

输入：`右边车窗未关闭`

语义理解：`未关闭 = 打开`

展开结果：
1. `右前车窗打开`
2. `右后车窗打开`

**示例 7：前排概括**

输入：`前排座椅通风开启`

展开结果：
1. `左前座椅通风开启`
2. `右前座椅通风开启`

**示例 8：后排概括**

输入：`关闭后排阅读灯`

展开结果：
1. `左后阅读灯关闭`
2. `右后阅读灯关闭`

##### 7.3.4.6 实现原则

范围展开模块需要遵循以下原则：

1. 先识别对象，再识别范围，再识别动作。
2. 展开后的子项必须保留原始上下文，便于追踪。
3. 展开结果是候选语义，不代表最终一定有对应信号。
4. 如果信号库中不存在某个展开项，不能强行输出，后续由候选召回和模型判断决定。
5. 同一条用例可同时包含多个范围词和多个对象词，需要支持拆解。

##### 7.3.4.7 后续可扩展方向

后续可以继续补充更复杂的方位和区域表达，例如：

1. `驾驶侧`
2. `乘员侧`
3. `车头侧`
4. `车尾侧`
5. `内侧`
6. `外侧`

第一版先聚焦常见座舱/车身控制语义，确保实现稳定。

#### 7.3.5 否定式 / 反义表达

这是明确必须支持的规则。

示例：

1. `主驾座椅加热未打开` → `主驾座椅加热关闭`
2. `空调不处于开启状态` → `空调关闭`
3. `左前车窗未关闭` → `左前车窗打开`
4. `不是开启状态` → `关闭`
5. `未使能` → `关闭 / 禁用`

#### 7.3.6 否定词清单初稿

1. `未`
2. `不`
3. `不是`
4. `无`
5. `没有`
6. `未处于`
7. `不处于`
8. `未进入`

#### 7.3.7 反义映射规则

1. `未打开` → `关闭`
2. `未开启` → `关闭`
3. `未接通` → `断开`
4. `未关闭` → `打开`
5. `未锁止` → `解锁`
6. `未解锁` → `锁止`
7. `未使能` → `禁用`
8. `不是激活状态` → `未激活`

#### 7.3.8 注意事项

1. 第一版优先支持车身控制、座舱控制场景常见语义。
2. 对双重否定、复杂条件句、上下文依赖句先不做强规则求值，交给模型。
3. 规则增强的作用是：
   - 扩展候选召回
   - 为提示词提供辅助解释
   - 提高模型判断稳定性

#### 7.3.9 枚举值语义匹配

这是必须单独设计的一类能力。测试用例中的目标值描述，很多时候不会与信号中的枚举值说明完全一致，系统不能只按字面字符串相等匹配，而需要结合语义、等级关系和当前枚举集合做理解。

##### 7.3.9.1 典型问题

第一版至少需要覆盖以下情况：

1. 测试用例写自然语言，信号枚举写英文或代码化枚举名。
2. 测试用例写"2档"，信号枚举写 `Level2`。
3. 测试用例写"最大/最高/最强"，信号枚举是 `Level1-Level3` 或 `Low/Medium/High`。
4. 测试用例写"中档/高档/低档"，信号枚举是 `Middle/High/Low`。
5. 测试用例写"最小"，但信号中同时存在 `Off` 和 `Level1`，需要结合语义理解。

##### 7.3.9.2 典型示例

**示例 1：档位描述与枚举名不一致**

测试用例：`空调风量调到2档`

信号枚举：
```json
{"0": "Off", "1": "Level1", "2": "Level2", "3": "Level3"}
```
需要理解为：`2档 = Level2`，最终应输出：`signal_val = 2`

**示例 2：模糊上限描述**

测试用例：`空调风量调到最大`

信号枚举：
```json
{"1": "Level1", "2": "Level2", "3": "Level3"}
```
需要理解为：`最大 = 当前枚举集合中的最高等级 = Level3`，输出：`signal_val = 3`

**示例 3：中文等级与英文枚举**

测试用例：`座椅加热调到高档`

信号枚举：
```json
{"0": "Off", "1": "Low", "2": "Middle", "3": "High"}
```
需要理解为：`高档 = High`

**示例 4：中间档位**

测试用例：`风量保持中等`

信号枚举：
```json
{"1": "Low", "2": "Medium", "3": "High"}
```
需要理解为：`中等 = Medium`

**示例 5：最小值的歧义**

测试用例：`风量调到最小`

信号枚举：
```json
{"0": "Off", "1": "Level1", "2": "Level2", "3": "Level3"}
```

这里不能机械理解为最小数值。通常需要优先理解为"最低有效档位"，即 `最小 = Level1`。但如果上下文明确表示关闭，则也可能对应 `Off`。

第一版原则：
1. `最小` 优先映射为最低有效档位。
2. 如果上下文是"关闭/未打开/关到最小"，则允许模型结合枚举判断是否为 `Off`。
3. 如果无法确定，不允许猜测。

##### 7.3.9.3 第一版必须覆盖的值语义类型

1. 档位类：`1档 / 一档 / 一级`、`2档 / 二档 / 二级`、`3档 / 三档 / 三级`
2. 高低等级类：`高`、`中`、`低`、`高档`、`中档`、`低档`
3. 上下限类：`最大`、`最小`、`最高`、`最低`、`最强`、`最弱`
4. 英文枚举类：`Low`、`Medium`、`Middle`、`High`、`Level1`、`Level2`、`Level3`、`Min`、`Max`
5. 状态值类：`开启`、`关闭`、`使能`、`禁用`、`激活`、`未激活`

##### 7.3.9.4 实现原则

枚举值匹配建议分三层处理：

1. 字面匹配：用例描述直接命中枚举说明
2. 规则归一化匹配：`2档 → Level2`、`高档 → High`、`中档 → Medium/Middle`
3. 枚举集合语义匹配：`最大 → 当前枚举集合中的最高等级`、`最小 → 当前枚举集合中的最低有效等级或关闭（需结合上下文）`

##### 7.3.9.5 不允许猜测的场景

以下情况必须谨慎处理，必要时返回"值不明确"：

1. 枚举值只有数字，没有任何说明，但测试用例写"最大"。
2. 枚举关系无法确定高低顺序。
3. 测试用例写"中等偏高""稍强一点"等模糊描述，而枚举定义不足以支撑精确映射。

##### 7.3.9.6 对候选召回和 Prompt 的影响

枚举值语义匹配不仅影响最终 `signal_val`，也影响候选召回和 Prompt 设计：

1. 候选召回时要把值语义一并纳入，例如：
   - `2档 → 2 / 二档 / Level2`
   - `最大 → Max / High / 最高等级`
2. Prompt 中必须提供完整枚举说明。
3. Prompt 中必须明确告诉模型：测试用例中的值描述可能与枚举说明不完全一致，需要在提供的枚举集合中寻找语义最接近的目标值。

### 7.4 候选信号召回模块

#### 目标

在把信号发给模型前，先从全量信号库中召回一小批高相关候选，减少 token 消耗并提高匹配准确率。

#### 召回依据

1. 用例原文关键词
2. 归一化后的动作词
3. 归一化后的位置词
4. 范围展开后的多个子句
5. 否定式转换后的目标状态词
6. 信号描述字段
7. 信号名
8. 枚举值说明
9. 报文名

#### 7.4.1 枚举值相关召回补充

候选召回不能只匹配控制对象，也要尽量把目标值语义带入召回上下文。例如：

1. `空调风量调到2档`
   需要把 `2档` 同时扩展为：`2`、`二档`、`Level2`
2. `风量调到最大`
   需要把 `最大` 扩展为：`最高`、`Max`、`High`、当前候选枚举中的最高等级项

这样可以提高模型对目标信号值的判断准确率。

#### 召回策略

第一版采用规则召回：

1. 从 `case_step` 提取关键词。
2. 用同义词词典扩展关键词。
3. 对范围描述进行展开。
4. 在扁平化信号列表中做打分匹配。
5. 取 TopN 候选信号给模型。

#### 召回结果示例

```json
{
  "case_id": "tc_001",
  "expanded_steps": [
    "打开主驾座椅加热",
    "开启左前座椅加热"
  ],
  "candidate_signals": [
    {
      "signal_name": "DrHeatSts",
      "signal_desc": "主驾加热状态",
      "msg_id": "0x22A",
      "values": {
        "0": "OFF",
        "1": "ON"
      }
    }
  ]
}
```

#### 后续增强

后续可升级为：

1. 规则召回 + BM25
2. 规则召回 + 向量检索

第一版先不引入向量库。

### 7.5 Prompt 构建模块

#### 目标

构建高约束系统提示词与用户提示词，确保模型只在已知信号信息中做语义匹配，不编造。

#### 7.5.1 系统提示词必须强调的规则

1. 你只能从提供的信号信息中查找匹配项。
2. 不允许虚构信号名、报文 ID、信号值、信号描述。
3. 测试用例可能是自然语言变体，不能只按字面匹配。
4. 需要理解以下语义现象：
   - 顺序颠倒
   - 动作同义词
   - 位置别名
   - 范围展开
   - 全量描述展开
   - 一条用例对应多个信号
   - 否定式 / 反义表达
5. 如果信号信息中没有足够证据，必须输出未匹配，不允许猜测。
6. 输出必须为严格 JSON。

#### 7.5.2 提示词中必须包含的自然语言说明

必须明确写入以下规则：

1. `主驾` 等价于 `左前`
2. `副驾` 等价于 `右前`
3. `左侧` 一般表示 `左前 + 左后`
4. `右侧` 一般表示 `右前 + 右后`
5. `全部车窗` 表示四门车窗全部展开
6. `未打开` 通常表示 `关闭`
7. `未关闭` 通常表示 `打开`
8. `不是开启状态` 通常表示 `关闭`

#### 7.5.3 用户提示词组成

每次模型调用的用户提示词应包含：

1. 当前要解析的用例
2. 当前用例的归一化语义说明
3. 当前用例召回到的候选信号列表
4. 目标输出 JSON schema
5. 示例输出

#### 7.5.4 Few-shot 示例要求

必须至少包含：

1. 单信号直接匹配
2. 多信号展开匹配
3. 主驾 / 左前同义示例
4. 左侧 / 右侧范围展开示例
5. 全部车窗展开示例
6. 否定式 / 反义表达示例
7. 枚举值语义匹配示例
8. 未匹配示例

### 7.7 模型输出标准化模块

这是另一个核心模块。

#### 目标

使用 `Pydantic` 对模型输出进行强约束，确保后续导出和回填稳定。

#### 7.7.1 设计原则

1. 模型返回必须是结构化 JSON。
2. 后端必须对返回结果做 schema 校验。
3. 对常见问题做容错，按以下优先级尝试修复后再校验：
   - 去除 markdown 代码块包裹（` ```json ... ``` `）
   - 去除响应前后多余空白
   - 提取第一个合法 JSON 对象或数组
   - 如修复后仍校验失败，记录错误并标记该条用例为校验失败状态，不中断整体任务

#### 7.7.2 Pydantic 模型草案

```python
from pydantic import BaseModel, Field
from typing import List, Optional


class SignalCandidate(BaseModel):
    signal_desc: Optional[str] = Field(default=None, description="信号描述")
    msg_id: Optional[str] = Field(default=None, description="报文ID，优先十六进制字符串")
    signal_name: str = Field(description="信号名")
    signal_val: Optional[str] = Field(default=None, description="目标信号值")
    info_str: Optional[str] = Field(default=None, description="格式化后的汇总字符串")
    match_reason: Optional[str] = Field(default=None, description="匹配原因")


class CaseMatchResult(BaseModel):
    case_id: str = Field(description="用例ID")
    case_step: str = Field(description="原始用例步骤")
    matched: bool = Field(description="是否匹配成功")
    case_info: List[SignalCandidate] = Field(default_factory=list, description="匹配到的信号列表")
    unmatched_reason: Optional[str] = Field(default=None, description="未匹配原因")


class MatchResponse(BaseModel):
    task_id: str = Field(description="任务ID，便于后续查询")
    results: List[CaseMatchResult] = Field(default_factory=list, description="匹配结果列表")
```

#### 7.7.3 标准输出示例

```json
{
  "task_id": "task_20240101_001",
  "results": [
    {
      "case_id": "tc_001",
      "case_step": "打开主驾座椅加热",
      "matched": true,
      "case_info": [
        {
          "signal_desc": "主驾加热状态",
          "msg_id": "0x22A",
          "signal_name": "DrHeatSts",
          "signal_val": "1",
          "info_str": "【0x22A, DrHeatSts, 1】",
          "match_reason": "用例描述表示主驾座椅加热开启，候选信号中主驾加热状态与其匹配，枚举值1表示ON"
        }
      ],
      "unmatched_reason": null
    }
  ]
}
```

#### 7.7.4 未匹配示例

```json
{
  "task_id": "task_20240101_001",
  "results": [
    {
      "case_id": "tc_002",
      "case_step": "关闭方向盘加热",

      "case_info": [],
```

## 10. Prompt 设计草案

### 10.1 系统提示词设计目标

让模型明确知道：

1. 它不是在自由生成答案，而是在提供的信号库中做严格匹配。
2. 它必须理解车控语义常见自然语言表达。
3. 它必须输出严格 JSON。

### 10.2 系统提示词关键规则清单

系统提示词中至少要包含以下信息：

1. 只能从候选信号中查找匹配项。
2. 禁止编造。
3. 一条用例可以对应多个信号。
4. 若信号值枚举明确，必须返回正确值。
5. 若枚举不明确，不能猜测。
6. 要理解常见自然语言等价关系：
   - `主驾 = 左前`
   - `副驾 = 右前`
   - `左侧 = 左前 + 左后`
   - `右侧 = 右前 + 右后`
   - `全部车窗 = 四门车窗`
7. 要理解反义表达：
   - `未打开 = 关闭`
   - `未关闭 = 打开`
   - `不是开启状态 = 关闭`
   - `不处于激活状态 = 未激活`
8. 信号找不到时必须明确返回未匹配原因。
9. 要理解测试用例中的目标值描述不一定与信号枚举说明逐字一致，例如：
   - `2档 = Level2`
   - `高档 = High`
   - `最大 = 当前枚举集合中的最高等级`
10. 对于"最大/最小/最高/最低"这类模糊等级词，必须结合当前信号提供的枚举集合判断，不能脱离枚举上下文编造值。

1. 当前用例原始文本
2. 归一化说明
3. 拆解后的语义子任务
4. 候选信号列表
5. 候选信号的枚举值说明
6. 输出 JSON 模板

## 12. 错误处理策略

### 12.1 信号文件相关错误

1. 文件类型不支持
2. Excel 未识别到关键列
3. DBC 解析失败
4. 文件为空

### 12.2 测试用例文件相关错误

1. 未识别到用例 ID 列
2. 未识别到用例步骤列
3. 表格为空

### 12.3 模型调用相关错误

1. API Key 无效
2. 接口超时
3. DeepSeek 返回非 JSON
4. DeepSeek 返回字段不完整

### 12.4 结果处理错误

1. 结构化校验失败
2. 导出 Excel 失败

### 12.5 错误返回原则

1. 前端展示用户可理解错误（不暴露堆栈信息）
2. 后端保留详细日志（含完整堆栈）
3. 对模型输出结构错误按以下顺序自动修复后重新校验：
   - 去除 markdown 代码块包裹（` ```json ... ``` `）
   - 去除响应前后多余空白与控制字符
   - 提取响应中第一个合法 JSON 对象
   - 修复后仍失败则记录错误，标记该条为校验失败，不中断整体任务
4. 超时错误自动重试最多 2 次，仍失败则返回超时错误码

## 14. 数据库表结构设计

### 14.1 设计目标

第一版数据库设计的核心目标不是复杂业务建模，而是保证整条处理链路可追踪、可排查、可复盘。

数据库需要支持以下场景：

1. 查看一次任务的完整执行过程
2. 对比提示词修改前后的效果
3. 对比代码修改前后的召回和匹配结果
4. 定位是解析问题、召回问题、Prompt 问题还是模型输出问题
5. 保留模型原始响应，便于问题复盘

---

### 14.2 第一版数据库选型

第一版建议：

1. 开发环境使用 `SQLite`
2. 表结构设计兼容后续迁移到 `PostgreSQL`

原因：

1. `SQLite` 上手快，便于本地开发
2. 当前项目以本地单任务处理为主
3. 后续如果要支持多用户、多任务、复杂检索，可平滑迁移到 `PostgreSQL`

建议 ORM/数据库访问方案：

1. `SQLAlchemy 2.x`
2. 配合 `Alembic` 做迁移管理（首次建表和后续字段变更均通过 Alembic migration 执行，不使用 `create_all()` 直接建表）

---

### 14.3 设计原则

1. 原始数据与标准化数据都尽量保留
2. 每一条用例的每一步处理过程都尽量可追踪
3. 大字段优先使用 JSON 文本或 JSON 类型存储
4. 模型原始请求与原始响应必须可回溯
5. 数据库结构优先服务排错与调优，而不是先做性能极致优化

---

### 14.4 核心实体关系

第一版建议围绕一个"匹配任务"建立全链路数据模型。

核心关系如下：

1. 一个 `match_task` 对应一次完整执行
2. 一个 `match_task` 对应一个信号源文件
3. 一个 `match_task` 对应一个测试用例文件
4. 一个 `match_task` 包含多条测试用例
5. 每条测试用例可以有：
   - 一条归一化记录
   - 多条候选召回记录
   - 一条或多条模型调用记录
   - 一条结构化匹配结果记录
6. 一个任务最终可生成一个导出文件记录

3. 看 `case_candidate_signal` — 候选召回是否正确
4. 看 `prompt_record` — Prompt 是否把必要上下文传给模型

## 16. 当前确认的关键原则

1. `Pydantic` 是模型输出标准化的核心，不是可选项。
2. 模型只能从提供的信号信息中匹配，不允许虚构。
3. 提示词必须显式说明各种常见自然语言表达，不可依赖模型自行猜测。
4. 一条用例可能对应多个信号。
5. 否定式 / 反义表达必须纳入规则和提示词，例如：
   - `未打开 = 关闭`
   - `未关闭 = 打开`
6. 第一版必须做数据库持久化，保证全过程留痕。
7. `api_key` 等敏感信息通过 `pydantic-settings` + `.env` 文件管理，不在请求体中明文传输，不硬编码。
8. 模型调用失败时自动重试，模型输出结构异常时尝试自动修复后再校验，均需落库留痕。
9. 数据库表结构通过 Alembic migration 管理，保证可追溯和可迁移。
